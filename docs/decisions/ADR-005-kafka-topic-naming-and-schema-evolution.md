# ADR-005: Kafka 토픽 네이밍 및 스키마 진화 정책

- **상태**: 확정
- **결정**: 토픽 명은 `{domain}.{event-name}.{version}` 형식. 페이로드는 JSON. 스키마 진화는 backward compatibility만 보장(필드 추가만 허용, 제거·타입 변경 금지). 파티션 키는 도메인 애그리거트 ID(`orderId`).

## 근거

- **명명 규칙 `{domain}.{event-name}.{version}`**:
  - README가 이미 `order.order-confirmed.v1` 형태를 예시로 제시 — 같은 컨벤션을 따르면 토픽 이름만으로 어느 도메인의 어느 이벤트인지 즉시 파악 가능.
  - 도메인 prefix는 다중 서비스가 공용 클러스터를 쓸 때 충돌 방지.
  - 마지막의 `vN`은 **토픽 자체에 버전 박기**. 호환되지 않는 변경(필드 제거·의미 변경)이 필요할 때 새 토픽(`v2`)을 만들고 기존 토픽은 발행 중단 후 폐기 — 컨슈머가 점진 이행 가능.
- **JSON 페이로드**:
  - Spring for Apache Kafka 기본 직렬화 경로가 JSON(`JsonSerializer`/`JsonDeserializer`)이라 추가 의존성 없이 즉시 사용.
  - Avro/Protobuf는 schema registry 운영 비용 동반. 본 과제는 Embedded Kafka + 단일 컨슈머 시나리오라 schema registry 도입의 가치가 거의 없음.
- **Backward compatibility만 보장**:
  - "**필드 추가만 허용**"이라는 단순 규칙으로 운영 비용 최소화. 발행자가 새 필드를 추가해도 기존 컨슈머는 무시.
  - Forward compatibility(컨슈머가 모르는 필드 처리)까지 보장하려면 디시리얼라이저 설정·테스트 매트릭스가 추가되는데, 본 과제 범위에서 이를 검증할 컨슈머가 없어 ROI 부족.
- **파티션 키 = `orderId`**:
  - 동일 주문에 대한 이벤트(예: `order.order-confirmed.v1`, 보상 흐름의 `inventory.release-requested.v1`)가 동일 파티션으로 라우팅되어 **컨슈머 측에서 순서 보장**.
  - 다른 주문 간에는 병렬 처리 가능 → 처리량 손실 없음.

## 검토한 대안

### 명명 — `{event-name}` 단순 형식 (도메인·버전 prefix/suffix 없이)
- 기각 사유: 도메인 충돌 방지 불가. 버전을 토픽에 박지 않으면 호환 깨짐 변경이 필요할 때 컨슈머 점진 이행이 불가능.

### 명명 — `{service}.{event-name}` (도메인 대신 서비스 명)
- 기각 사유: 발행 서비스가 변경(분리·병합)될 때 토픽 이름이 의미를 잃음. 도메인은 서비스보다 안정적이라 이쪽이 우월.

### 페이로드 — Avro + Schema Registry
- 기각 사유: schema registry 컨테이너 추가 + Avro 클래스 빌드 단계 추가 = 운영·빌드 복잡도. 본 과제 핵심 영역(Saga·통신·성능)에 직접 기여 없음.

### 스키마 진화 — full compatibility (forward + backward)
- 기각 사유: 필요한 검증 매트릭스가 비대(구버전↔신버전 양방향 페이로드 호환 테스트). 본 과제에 실제 컨슈머가 부재해 검증 표면 자체가 없음.

### 파티션 키 — 라운드 로빈(키 없음)
- 기각 사유: 동일 주문의 이벤트가 다른 파티션에 분산되면 보상·후속 처리 시 순서 보장이 깨짐. Saga의 멱등 키만으로 회복 가능하지만, 키 한 줄 지정으로 회피 가능한 비용을 굳이 멱등성으로 메우는 건 우회적.

## 검증과 한계

- **검증**:
  - 통합 테스트(`@EmbeddedKafka`)에서 발행된 메시지의 토픽명·키·페이로드 구조를 assert.
  - 페이로드 클래스에 필드를 하나 추가한 변형 테스트로 backward compatibility 회귀 검증(컨슈머가 신규 필드를 무시하고 정상 처리).
- **한계**:
  - Schema registry 미사용으로 **타입 변경·필드 제거 같은 호환 깨짐 변경을 빌드 시점에 막지 못함**. 정책으로만 강제 — 실제 운영에서는 registry나 contract 테스트 도입 필요. README "의도적으로 하지 않은 것"에 명시.
  - 파티션 키 `orderId`로 해시 분산 시 특정 주문에 트래픽이 집중되면 핫 파티션 위험. 본 과제 트래픽 모델(주문당 단 몇 건의 이벤트)에서는 비현실적이지만, 장기 운영 시 모니터링 필요.
